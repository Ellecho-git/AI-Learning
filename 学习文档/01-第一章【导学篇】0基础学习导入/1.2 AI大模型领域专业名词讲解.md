## 第2节 AI大模型领域专业名词讲解

### 📌 本节核心
- 四个基础概念：提示词、Token、上下文长度、模型参数量
- 三种模型优化技术：模型蒸馏、Function Calling、MCP协议
- AI Agent的构成要素：大模型 + 记忆 + 规划能力 + 工具使用

---

### 📖 详细笔记

#### 一、基本概念

##### 1. 提示词 (Prompt)
- **定义**：用户与模型交互的入口，即输入给模型的指令或问题
- **作用**：提示词的质量直接影响模型输出的效果
- **分类**：
  | 类型 | 说明 | 示例 |
  |:---|:---|:---|
  | 用户提示词 | 最终用户直接输入的内容 | “帮我写一封求职邮件” |
  | 系统提示词 | 开发者设定的底层指令，用于定义模型的角色和行为 | “你是一位资深HR，擅长修改简历” |
- **限制**：模型对文本长度有限制，提示词过长可能会被截断

##### 2. Token (词元)
- **定义**：模型处理文本的基本单位。输入文字时，模型会将其切分成若干个 token 进行计算
- **关键点**：
  | 维度 | 说明 |
  |:---|:---|
  | 计费方式 | 大多数 API 按“输入 token 数 + 输出 token 数”计费 |
  | 语言差异 | 不同语言、甚至不同模型对同一段文本的 token 切分方式不同。中文一个字可能被切分成 1-3 个 token |
  | 效率问题 | 选择对中文更“友好”的模型，可以节省 token 消耗 |

##### 3. 上下文长度
- **定义**：模型单次能处理的最大文本长度，以 token 为单位
- **常见规格**：16K、32K、128K、1M 等
- **直观感受**：
  | 规格 | 对应文本量 |
  |:---|:---|
  | 16K | ≈ 8千~1.1万字（约20页 Word 文档） |
  | 128K | ≈ 6.4万~9万字（《小王子》全书约 6万字） |
  | 1M | ≈ 可以一次性处理三本《三体》 |
- **注意**：系统提示词会占用一部分上下文空间，实际可用长度会略少

##### 4. 模型参数量
- **定义**：模型参数的数量，通常以“B”（Billion，十亿）为单位，如 7B、14B、32B、671B
- **特点**：参数量越大，模型能力越强，但这是一把双刃剑
  | 维度 | 说明 |
  |:---|:---|
  | 优势 | 模型更聪明，理解能力更强 |
  | 代价 | 部署成本高，如 DeepSeek 671B 满血版部署一次需要 200 万+ |
  | 限制 | 计算性能要求高，无法跑在手机等低算力设备上 |
- **权衡**：实际应用中需在“能力”和“成本”之间做权衡

#### 二、模型优化与应用

##### 1. 模型蒸馏 (Model Distillation)
- **定义**：把大模型（老师）的知识“传授”给小模型（学生）的过程
- **过程**：学生模型在学习过程中模仿老师的行为，最终在保持较高水平的同时，大幅缩小模型体积
- **适用场景**：手机端、智能音箱、可穿戴设备等算力有限的场景

##### 2. 功能调用 (Function Calling)
- **定义**：让模型具备调用外部工具的能力
- **解决的问题**：模型的“知识截止”问题（无法获取实时信息）
- **示例对比**：
  | 场景 | 表现 |
  |:---|:---|
  | 用户问 | “明天北京适合穿什么？” |
  | 传统模型 | 只能根据训练数据泛泛而谈 |
  | 支持 Function Calling 的模型 | 调用天气 API → 获取实时预报 → 给出精准建议 |

##### 3. MCP协议 (Model Context Protocol)
- **定义**：一个标准化工具调用的协议
- **产生背景**：传统的 Function Calling 存在以下问题
  - 每个工具都需要单独开发接口
  - 切换模型时需要重新配置
  - 权限控制、错误处理都要重复做
- **组成部分**：
  | 组件 | 作用 |
  |:---|:---|
  | MCP Server | 提供具体功能的服务程序 |
  | MCP Tool | Server 提供的具体工具（如查天气、订会议室） |
  | MCP Client | 调用工具的 AI 模型或应用 |
- **优势**：
  - ✅ 开发一次，多处复用
  - ✅ 换模型不用重配
  - ✅ 实时通信，安全可控

#### 三、智能体 (AI Agent)

##### 1. 定义
- **AI Agent**：一种能够感知环境、自主决策并执行动作的智能实体
- **与传统 AI 的区别**：传统 AI“回答问题”，AI Agent“完成任务”

##### 2. 构成要素
- **经典公式**：AI Agent = 大模型 + 记忆 + 规划能力 + 工具使用
- **要素详解**：
  | 要素 | 作用 | 类比 |
  |:---|:---|:---|
  | 大模型 | 负责理解和决策 | 大脑 |
  | 记忆 | 存储上下文和历史信息 | 备忘录 |
  | 规划能力 | 拆解任务、制定步骤 | 项目经理 |
  | 工具使用 | 调用 API、执行操作 | 执行者 |

##### 3. 一个完整的Agent工作流程
- **任务示例**：“分析公司情况并生成报告和可视化网页”
- **执行过程**：
  | 步骤 | 动作 |
  |:---|:---|
  | 任务规划 | 拆解为“查资料 → 分析数据 → 写报告 → 生成网页” |
  | 工具调用 | 调用搜索引擎、代码生成工具 |
  | 记忆存储 | 记录已查到的信息，保持上下文连贯 |
  | 输出结果 | 一份分析报告 + 一个可视化网页 |
- **特点**：整个过程不需要人为干预，Agent 自主完成

##### 4. 低代码开发平台
- **用途**：快速体验 AI Agent，无需从零写代码
- **常见平台**：
  - Coze（扣子）
  - 文心智能体平台
  - 腾讯元器
  - 阿里云百炼
- **核心功能**：选一个模型、配点记忆、拖拽一个工作流，即可搭建可用 Agent

#### 四、学习小结
- **基础层**：提示词（怎么问）→ Token（怎么计费）→ 上下文（能记多少）→ 参数量（有多聪明）
- **优化层**：蒸馏（变小）→ Function Calling/MCP（变能干）
- **应用层**：AI Agent = 大模型 + 记忆 + 规划 + 工具

### 💡 总结
大模型核心概念 = 基础词（知道怎么聊） + 计费逻辑（知道怎么省） + 优化手段（知道怎么改） + 高级玩法（知道怎么用）
